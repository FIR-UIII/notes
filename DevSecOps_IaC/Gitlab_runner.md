### Как работает GitLab Runner

Общая архитектура
```
┌─────────────────┐        ┌─────────────────┐        ┌─────────────────┐
│   GitLab Server │  ◄───► │  GitLab Runner  │  ───►  │  Executor       │
│  (CI/CD Jobs)   │  poll  │  (агент)        │ запуск │  (shell/docker) │
└─────────────────┘        └─────────────────┘        └─────────────────┘
```

GitLab Runner — это агент, который устанавливается на отдельную машину (или контейнер) и постоянно опрашивает GitLab Server на наличие новых задач (jobs). Когда появляется job:

1. Runner забирает job через GitLab API
2. Скачивает репозиторий (git clone / fetch)
3. Запускает job через выбранный executor
4. Стримит логи обратно в GitLab в реальном времени
5. Возвращает результат (success / failed)

Runner не ждёт push от GitLab — он сам делает polling каждые несколько секунд (по умолчанию 3 сек)

---
Типы executors
```
[[runners]]
executor = "shell"
```
1. Shell
executor = "shell"
Job выполняется напрямую в shell той машины, где установлен runner (bash, sh, PowerShell).

Как работает:
- Скрипт из .gitlab-ci.yml запускается как обычный bash-процесс
- Все инструменты (Node, Python, Docker, etc.) должны быть установлены на машине заранее
- Рабочая директория: /home/gitlab-runner/builds/...

Плюсы:
- Максимальная скорость — нет overhead на создание контейнера
- Доступ к GPU, железу, локальным сервисам
- Кэш между запусками работает нативно

Минусы:
- Нет изоляции — один job может сломать окружение другого
- Зависимости нужно ставить вручную на каждую машину
- Безопасность: вредоносный .gitlab-ci.yml имеет полный доступ к машине

Когда использовать: сборка embedded/firmware, legacy-системы, доступ к физическому железу, когда скорость критична.

---
2. Docker
```
[[runners]]
executor = "docker"
[runners.docker]
image = "ubuntu:22.04"
```

Каждая job запускается в свежем Docker-контейнере.

Как работает:
1. Runner поднимает контейнер из образа, указанного в image:
2. Клонирует репозиторий внутрь контейнера
3. Выполняет скрипты
4. Контейнер уничтожается после завершения job

```
# .gitlab-ci.yml
build:
  image: node:20-alpine     # ← какой контейнер использовать
  script:
    - npm ci
    - npm run build

Services — дополнительные контейнеры (БД, Redis и т.д.), которые запускаются рядом:
test:
  image: python:3.12
  services:
    - postgres:15       # ← отдельный контейнер, доступен по hostname "postgres"
    - redis:7
  script:
    - pytest
```

Плюсы:
- Полная изоляция между jobs
- Гибкость: разные jobs — разные образы
- Воспроизводимость: образ одинаков везде
- Нет "мусора" после job

Минусы:
- Медленнее shell из-за запуска контейнера (~5–15 сек overhead)
- Docker-in-Docker (dind) для сборки образов требует привилегий или специальной настройки
- Кэш node_modules / pip сложнее настроить

Когда использовать: большинство CI/CD пайплайнов — это стандартный выбор.

---
3. Docker Machine (autoscaling)
```
[[runners]]
  executor = "docker+machine"
```
Расширение Docker executor: runner автоматически создаёт виртуальные машины (через Docker Machine) под каждый job и удаляет их после завершения.

Как работает:
- Нет нагрузки — нет VM, платить не нужно
- Пришёл job — runner поднял VM в облаке (AWS EC2, GCP, Azure), запустил Docker-контейнер внутри VM, выполнил job, удалил VM

Когда использовать: переменная нагрузка, облачные окружения, нужна горизонтальная масштабируемость. Deprecated в пользу autoscaler на Fleeting.

---
4. Kubernetes
```
[[runners]]
  executor = "kubernetes"
  [runners.kubernetes]
    namespace = "gitlab-runner"
    image = "ubuntu:22.04"
```

Каждый job запускается как отдельный Pod в Kubernetes-кластере.

Как работает:
1. Runner создаёт Pod через Kubernetes API
2. Pod содержит несколько контейнеров: helper (клонирует репо), build (выполняет job), service-контейнеры
3. После завершения Pod удаляется

```
# .gitlab-ci.yml — можно переопределить ресурсы
build:
  image: golang:1.22
  tags:
    - kubernetes
  variables:
    KUBERNETES_CPU_REQUEST: "500m"
    KUBERNETES_MEMORY_LIMIT: "1Gi"
```

Плюсы:
- Нативный autoscaling через Kubernetes
- Изоляция уровня Pod
- Удобно, если инфраструктура уже на K8s
- Limits/requests для ресурсов из коробки

Минусы:
- Сложная настройка RBAC, PVC для кэша
- Docker-in-Docker требует дополнительной настройки (kaniko или dind)
- Запуск Pod медленнее, чем контейнер

Когда использовать: production CI на Kubernetes-инфраструктуре, большие команды с переменной нагрузкой.

---
5. VirtualBox / Parallels
```
[[runners]]
  executor = "virtualbox"
```
Job запускается в виртуальной машине (VirtualBox или Parallels). Runner клонирует базовый snapshot VM для каждого job.

Когда использовать: тестирование для macOS/Windows, когда нужна полная ОС, а не контейнер.

---
6. SSH
```
[[runners]]
  executor = "ssh"
  [runners.ssh]
    host = "192.168.1.100"
    user = "ci"
```

Runner подключается к удалённой машине по SSH и выполняет job там.

Когда использовать: сборка на удалённом железе (например, ARM-сервер, Mac mini для iOS сборок).

---
7. Custom Executor

Позволяет написать собственный executor на любом языке — набор bash-скриптов, которые Runner вызывает на определённых этапах (prepare, run, cleanup).
Когда использовать: нестандартные среды — Podman, LXC, Firecracker microVMs, собственный облачный провайдер.

Конфигурационный файл config.toml
```
concurrent = 4          # сколько jobs параллельно на этом runner

[[runners]]
  name = "my-docker-runner"
  url = "https://gitlab.com"
  token = "xxxx"
  executor = "docker"

  [runners.docker]
    image = "alpine:latest"
    volumes = ["/cache"]     # кэш между запусками
    privileged = false        # docker-in-docker требует true
    pull_policy = "if-not-present"

  [runners.cache]
    type = "s3"              # кэш в S3 для shared runners
    [runners.cache.s3]
      bucket_name = "my-ci-cache"
```

---
Итого: для большинства проектов выбирают Docker executor как баланс между изоляцией и простотой. Shell используют когда нужна скорость или доступ к железу. Kubernetes — когда инфраструктура уже построена на нём и нужен autoscaling.
