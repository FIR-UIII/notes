SPIFFE (произносится как "спифи") — это набор открытых стандартов для создания криптографически подтвержденных идентификаторов для программных рабочих нагрузок (микросервисов, контейнеров, виртуальных машин и т.д.) в динамических и гетерогенных средах
Он решает фундаментальную проблему: как один сервис может с уверенностью сказать, что другой сервис — это именно тот, за кого себя выдает, особенно в мире, где IP-адреса меняются, а общие ключи API трудно контролировать

SPIFFE ID (Имя) `spiffe://<trust-domain>/<workload-identifier>`
SVID (SPIFFE Verifiable Identity Document — Удостоверяющий документ). Он существует в двух основных форматах:
* X.509-SVID: Это стандартный цифровой сертификат. Он идеально подходит для установления взаимных TLS-соединений (mTLS) между сервисами.
* JWT-SVID: Это JSON Web Token (JWT). Он лучше подходит для HTTP/REST API контекстов, где идентичность нужно передавать в заголовках запросов

⚙️ Как это работает на практике: Роль SPIRE
SPIFFE — это стандарт, а SPIRE — это его эталонная, готовая к использованию в продакшене реализация. Именно SPIRE "оживляет" SPIFFE. 
Архитектура SPIRE состоит из двух основных компонентов:
```
SPIRE Server (Сервер): Это "мозг" и центр управления всей системой. Он:
  Управляет всеми идентичностями (SPIFFE ID) в своем домене доверия.
  Хранит правила (регистрационные записи), которые определяют, какие условия должны быть выполнены, чтобы выдать конкретный SPIFFE ID.
  Выступает в роли удостоверяющего центра (CA) и подписывает SVID.

SPIRE Agent (Агент): Этот компонент запускается на каждом узле (виртуальной машине или физическом сервере), где работают ваши сервисы. Он:
  Запрашивает SVID у SPIRE Server и кэширует их.
  Предоставляет SPIFFE Workload API локально для всех сервисов на этом узле.
  Проверяет (аттестует) личность рабочей нагрузки, которая пытается вызвать этот API.
```

🔄 Процесс получения идентичности
* Регистрация: Администратор регистрирует в SPIRE Server правило: "Любому процессу, запущенному в контейнере с образом billing-service и принадлежащему пользователю app-user, выдай SPIFFE ID spiffe://prod.acme.com/billing/api".
* Запрос: Ваш сервис (billing/api) запускается на узле и вызывает локальный SPIFFE Workload API, предоставляемый SPIRE Agent'ом, чтобы получить свой SVID.
* Аттестация (Проверка): SPIRE Agent не спрашивает пароль. Он проверяет среду сервиса: его PID, пользователя, от которого он запущен, метаданные контейнера и т.д.. Убедившись, что все соответствует зарегистрированному правилу, он одобряет запрос.
* Выдача: SPIRE Agent обращается к SPIRE Server за SVID и выдает сервису короткоживущий X.509 или JWT SVID (срок действия может составлять всего несколько часов).
* Использование: Теперь сервис billing/api может использовать свой SVID для аутентификации при общении с другим сервисом, например, inventory/api, через mTLS. Сервис inventory проверит SVID от billing, используя доверенный сертификат (Trust Bundle) от SPIRE Server.
* Обновление: SPIRE Agent автоматически отслеживает срок действия SVID и заменяет его на новый до истечения старого, полностью автоматизируя управление жизненным циклом удостоверений

Есть интеграция с keycloak (26.5.0 или выше) через федерацию, т.е. Keycloak может доверять и принимать SPIFFE JWT SVID (удостоверения личности), выданные внешней системой, например SPIRE, и использовать их для аутентификации OAuth-клиентов (приложений, сервисов) вместо классических client_id и client_secret
